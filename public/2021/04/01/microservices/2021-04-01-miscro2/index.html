Other - 认识微服务架构

软件架构

个人理解：软件架构是一种方法论，架构师乃是方法论、经验论的产出者，却是能够
在项目中发挥重要作用。总结出通用的方法论的人本就是稀有的存在。

软件架构（卡梅隆大学）：计算机系统的软件架构是构建这个系统所需要的一组结构，包括软件
元素、它们之间的关系以及两者的属性。

实质是应用程序的架构是将软件分解为元素和这些元素之间的关系

架构的重要性在于 非功能需求 也可以称为质量属性或者能力。非功能性需求决定应用
程序运行时的质量，比如可扩展性和可靠性。它们也决定了开发阶段的质量，包括可维护性、可
预测性、可扩展性和可维护性。

架构风格

建筑架构中架构风格限制了建筑的特征和建筑的材料。软件架构中架构风格决定了实例中使用的
组件和连接器的词汇表，以及如何组合它们的一种约束。

应用程序架构通常使用多种架构风格的组合。

微服务架构是将应用程序构造为一组松散耦合的服务

微服务架构风格

微服务的实现视图是由多个组件构成：一组可执行文件
组件即服务，每个组件可以有自己的架构风格
什么是服务？怎么理解松散耦合

服务：是一个单一的、可独立部署的软件组件，它实现了一些有用的功能
服务具有封装实现的 API 。API 定义了由客户端调用的操作。当服务的数据发生更改时，服务
会发布可供客户端订阅的事件。
松散耦合 是微服务最核心的特性。松散耦合服务是改善开发效率、提升可维护性和可测试性
例如：服务之间通信采用 API 完成，这样做的封装了服务的实现细节，允许在不影响客户端的情
况下，对实现修改、发布、部署。而且容易被理解

共享类库最好不要引入耦合。

连接器是指这些服务能够协作的通信协议

微服务的大小并不重要

如果因为其他服务的变更而不断需要更新自己负责的服务时，或者你所负责的服务变更触发其他服
务的同步更新，那么这表明服务没有实现松耦合。你构建的可能是一个分布式的单体

微服务的障碍

  • 网络延迟，进程间通信严重依赖网络
  • 通信降低了可用性
  • 需要维护跨服务之间的数据一致性

进程间通信

预防故障扩散

  • 正确的处理无响应服务
  • 需要决定如何从失败的远程服务中回复

服务发现

  • 服务实例的网络位置
    服务实例是动态分配的。自动扩展、故障和升级，服务集群实例就会更改

应用层服务发现模式

应用层实例服务发现的一种方式是应用程序的服务及其客户端与服务注册表进行交互
应用层多平台部署的方式好处就是可以处理多平台部署的问题。比如：Kubernetes
上只部署了一些服务其余的服务在 Kubernetes 之外，这种情况下注册中心就服务
发现机制就可以适用于这种环境。
弊端：为每个编程语言提供服务发现库

通常情况下可能需要负责设置和管理服务注册表，会分散一定的精力。所以最好的方
式是基础架构部部署基础设置提供服务发现机制。
