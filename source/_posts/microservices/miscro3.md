# 注册中心与服务发现的价值

服务发现的功能，就是在服务正在发起调用之前，调用方需要从注册中心拿到相应服务的可用 `IP:Port` 。服务发现
具有两种方式即 SDK 模式和 DNS 模式


## DNS

`DNS` 是域名解析系统，也是最早的提供注册中心和服务发现的系统。可以满足简单的服务发现场景，如双方约定好：
端口、序列化协议等。不可否认的就是所有的操作系统都实现了 `DNS` 客户端，所以 `DNS` 具有天然的跨语言的特性而
且，这也是接入微服务的最快的方式。`DNS` 作为服务发现大体分为两个步骤：

1. 将注册中心的数据以`DNS`的数据格式暴露出来
2. 任何系统的`DNS` 客户端都可以通过`DNS`协议获取服务列表

基于 DNS 服务的系统优点如下：

1. 集中的DNS服务器，便于升级维护
2. 面向开发友好，操作系统原生支持，跨语言天生支持

缺点：

1. 对DNS服务器性能要求高
2. 一般需要LVS设备为DNS服务器集群做请求转发，存在单点问题
3. 功能单一无法完成多元化的需求
4. 无法准确的获取服务占用的端口

## DNS Proxy

DNS Proxy 模式分为两种模式：
1. 本机 DNS 代理，拦截 DNS 请求做转发与服务发现
2. 应用程序中 DNS 代理，在应用程序中将 DNS 请求拦截做服务发现并返回

目前我们支持的注册中心是 Nacos ，Nacos 提供了一组简单易用的特性集，帮助您快速实现动态服务发现、服务
配置、服务元数据及流量管理。除了这些基本功能外还提供下面特性：

1. 主动和被动健康检查心跳上报功能，实时监控后端状态
2. 多种负载均衡策略，包括权重、同机房、同地域、同环境等
3. 多重容灾策略，比如：
   1. 服务端保护阈值，防止单点服务器压力过大宕机
   2. 客户端空列表保护，有效的防止误删 IP 操作或者服务端异常
4. 流量动态摘除，在后端某单点实例出现问题时，可以将流量摘除而不影响整体服务

这两种模式都是通过拦截本机域名解析请求后将请求转发至注册中心解析。在获取到服务的真是 IP 之后发起调用。
目前我们支持第二种模式，简单开发、减少运维，拦截域名解析需求可以很好的解决服务发现的问题，并且以后的
方向也是转向轻薄化 SDK 方向。处理 Nacos 注册中心提供以上的优点以外，客户端还有以下几点特性：

1. 客户端缓存策略，如果注册中心宕机客户端本地缓存可以提供服务，不影响正常调用
2. 降低开发复杂度，不用使用硬编码的网络地址，只需服务的名字就能使用服务
3. 易扩展，向后扩展方向可以完全掌控。比如将会支持：泳道、故障


比较与 Service 方式，Service 方式采用 ClusterIP 代理后端 IP 的方式，通常采用 Iptables 的
来处理流量的转发，iptables 通常使用三种模式：DNAT、Nat、Tunnel 三种模式。Iptables 本身执行方式
就是通过串行模式来执行，当服务实例达到一定规模之后会出现性能瓶颈，延迟的增加给用户带来一定的损失。
而且整个通信流程完全依赖于 ClusterIP 的转发，如果后端某一实例出现问题，无法准确的定位和流量摘除。

比较与 envoy 方式，envoy 是一个 Sidecar 系统，可以替代 SDN 大部分功能，但是 Envoy 存在一些如下
问题：
1. 相较于目前公司采用技术栈，有较高的门槛
2. 落地 envoy 使得运维成本指数上升
3. 如果完全依赖于 envoy 做服务发现， envoy 本身就存在稳定性问题
4. envoy 较耗时敏感的不友好，Envoy 本身就存在二次转发

当然微服务的发展方向倾向于 Service Mesh ，微服务的方向也将想 Envoy 靠拢，Envoy 可以提供如下功能：

1. 轻薄化 SDK
2. 提供集群限流、集群熔断功能简单
3. 无感升级，在轻薄化 SDK 的情况下 Envoy 升级相对简单，无需通知业务和运维即可原地升级
